# Password Generator — Technical Specification

> Статус: черновик v:2026.06-22-2 (обсуждаемый). Документ — источник истины для проекта.
> Если код или хотелка расходятся с ТЗ — расхождение обсуждается и фиксируется здесь, а не «по памяти».

---

## 1. Problem / Idea

Большинство генераторов выдают пароли, которые невозможно запомнить (`K7$mP2qx`). Их
записывают на бумажки (теряются) или в файл (теряется секретность). Известный обходной
путь — пароль из нескольких слов с разделителем (`correct-horse-battery-staple`): легко
запомнить, долго взламывать.

Идея проекта:

- Берём **2 слова по умолчанию** (настраивается) из словаря и разделитель.
- Слова пишутся русскими буквами, но **в английской раскладке**: `богема` → `,jutvf`.
  На экране — абракадабра (её трудно подсмотреть и запомнить с одного взгляда), а в голове
  пользователя — родное слово.
- **Честность — главная фича.** Инструмент не обещает невозможного: он прямо показывает
  реальную стойкость и подсказывает, как её повысить.

Это не «защита для Пентагона», а небольшой честный инструмент против бытовых угроз.

## 2. Threat model (от кого защищаем)

- **Главная угроза:** подглядывание через плечо / «диванный» хакер.
- **Вторичная:** тупой перебор по утёкшей базе — нейтрализуется уникальным паролем на сайт.
- **Вне зоны:** таргетированная атака с OSINT по соцсетям, кейлоггер на машине пользователя,
  принуждение («терморектальный криптоанализ»). Против гаечного ключа пароль бессилен — и
  инструмент это честно признаёт.
- **Принцип Керкгоффса:** считаем, что атакующий знает всё — схему, публичный словарь,
  раскладку, leet-список. Стойкость может давать **только настоящая неопределённость**,
  обфускация не считается.

## 3. Architecture

- Словарь отдается **как статика рядом со страницей**, единый файл-источник;
  приходит разовой загрузкой и живет в памяти браузера (никакой браузерной БД, никакой копии в бандле)
- Отказоустойчивость: генератор **работает в уже загруженной вкладке**.
  Первая загрузка при лежащем сервере -> ждем. После загрузки падение сервера/API отзывов генерацию не ломает
- Сервер имеет **две независимые роли**: статика (страница, бандл, словарь) и динамика (API отзывов, в будущем db) — падают раздельно

Клиент-серверное веб-приложение с разделёнными обязанностями. Два независимых канала.

- **Клиент (страница):** ядро на Kotlin/JS + тонкий UI на TS/HTML/SCSS. Выполняет **всю генерацию пароля в браузере**.
- **Сервер (Kotlin / Ktor):** отдаёт страницу, публичный словарь и обслуживает отзывы.
  **Пароли и секретные слова на сервер не попадают никогда.**

Два канала:

- **Канал паролей — односторонний.** Сервер → клиент: страница, словарь (`GET /api/words`,
  JSON). Обратно ничего секретного не уходит. Пароль/секреты живут только в браузере.
- **Канал отзывов — публичный, двусторонний.** Клиент ↔ сервер ↔ хранилище. По нему идёт
  **только публичный текст** (см. раздел 7), пароли и секреты сюда не попадают.

### Контракт данных словарь

- `/api/words` отдаёт JSON по data-классу `WordsResponse` (`version: String`, `words: List<String>`)
  из `commonMain`. Один класс — единственный источник правды для **формы** данных, общий для клиента и сервера.
- `kotlinx.serialization` — общая зависимость клиента (JS) и сервера (JVM).
- **Ядро генерации от JSON не зависит:** сериализация живёт только на границах (фетч на клиенте,
  эндпоинт на сервере). Алгоритм принимает уже готовый `List<String>`.

Почему генерация на клиенте:
- Пароль, прошедший через сервер, перестаёт быть секретным; секретные слова — тем более.

Ядро генерации: 
**Kotlin/JS (Multiplatform)** — общее ядро на Kotlin компилируется в JS,
тонкий UI на HTML/SCSS/TS.

### Два слоя фич (держать раздельно в коде и в индикаторе)

- **Слой стойкости** (считается в индикаторе): количество слов, личный секретный словарь.
- **Слой совместимости и памяти** (стойкости НЕ добавляет): раскладка, заглавные, leet, разделитель.

## 4. Input

- Количество слов (по умолчанию 2; настраивается 2–3).
- Разделитель (1 из 5 вариантов).
- Тумблеры совместимости: обеспечить заглавную / цифру / спецсимвол.
- Тумблеры стиля: leet (`про100`), капитализация.
- **Личные секретные слова:** список, который вводит пользователь. Живёт только в браузере.

## 5. Output

- Один пароль на экране (показываем открыто — его надо запомнить; на экране это
  абракадабра в раскладке).
- По умолчанию **2 слова** — «режим от соседа». **3 слова** доступен переключателем. Индикатор честно покажет, что против оффлайна это мгновенно
- Кнопка **«Сгенерировать ещё»** — прокрутить варианты и выбрать самый запоминаемый.
- **Копировать в буфер** + **автоочистка буфера через N секунд** (как KeePass).
- Опция «показать/скрыть» (блюр до наведения).
- Сгенерированные пароли **не сохраняются** на диск

### Честный индикатор стойкости — три части: 
  1. **Биты энтропии** — считаются только из настоящей случайности (см. §6, шаг 6). \
    Раскладка, leet, капитализация, фиксированный разделитель -> **0 бит**.
  2. **Время взлома — всегда с явно указанным допущением.** Без модели атакующего число
      бессмысленно, поэтому рядом пишем, *против кого* считаем. Два ориентира:
     - **Офлайн** (утекшая база, быстрый GPU, ~10^10 догадок/с) - основной honest-case;
     - **онлайн-throttled** (через форму с лимитом) - для контраста.
  3. **Вердикт с привязкой к threat model**, а не абстрактный:
    - от подглядываний через плечо — ок
    - от офлайн перебора — слабо

### Секретные слова (главный источник реального запаса)
- **Если есть секретное слово:** «посчитать нельзя — зависит от того, насколько ваше
  слово сложно угадать, и это хорошо». Не выдумываем число. 
- Можно показать **нижнюю границу** = биты только от публичной части, с явной пометкой
  «**+ неизвестный запас от секретных слов**». Не выдаем минимум за полную оценку. 
- **Честная рамка (ключевое):** только из публичного словаря пароль не дотягивает до «надежно» против Офлайна
  Реальный запас дает секретное слово. Индикатор это не прячет, а **ведет** к нему

Короткая подсказка про секретное слово (точный текст — на странице):
  - ✓ выдуманное слово · опечатка нарочно · детское словечко
  - ✗ кличка · группа · улица — это гуглят

## 6. Core Logic

1. Выбрать **N слов** (по умолчанию 2) источником случайности 
   **`crypto.getRandomValues()`** (Web Crypto; `Math.random()` запрещён) — из публичного и/или личного секретного словаря.
2. Применить маппинг **русская буква → английская раскладка** к каждому слову.
3. Соединить выбранным разделителем.
4. Опциональный стиль: капитализация / leet (из курируемого публичного leet-словаря).
5. **Проверка совместимости по ИТОГОВОЙ строке:** есть ли реальные `A–Z`, цифра, символ;
   если требуемого класса нет — добавить (в случайную позицию). Это важно, потому что
   капитализация в раскладке не всегда даёт букву (например `Х` → `{`, символ, а не буква).
6. **Оценка энтропии - только из настоящей неопределенности:**
   - публичные слова: `N_pub x log2(размер публичного словаря)`;
   - честные биты разделителя / цифры / символы — **только если они выбраны случайно**
     (фиксированный выбор = 0 бит)
   - раскладка, leet, капитализация = **0 бит** (детерминированны; по Керкгоффсу атакующий знает схему);
   - **секретное слово:** эффективный размер словаря неизвестен -> итоговым числом не показываем, 
     помечаем «+ неизвестный запас»; нужна нижняя оценка - берем только публичную часть и явно называем этот минимум;
   - **время взлома = f(биты, допущения об атакующем)**; допущение всегда показано рядом (см. §5, индикатор, часть 2)

> Транспорт/JSON — только на границах (см. §3, «Контракт данных»). В ядро приходит готовый
> `List<String>`; о JSON логика генерации не знает.

> Ориентир для калибровки вердикта (словарь ~250, офлайн ~10^10/с):
> 2 слова -> 250^2 = 62 тыс. = **6 мкс**; 3 слова -> 250^3 = 15,6 мл = **1,5 мс**.
> Оба – «мгновенно». Для сравнения `correct-horse` (4 x ~2048) = 44 бита.
> Вывод: 2 и 3 слова — это про «соседа», реальный запас дает только секретное слово.

## 7. Reviews / Feedback (коллективный разум)

Возможность писать отзывы/жалобы и показывать их. **Анонимно** (в духе секретности — имени и
PII не собираем). Показываем **последние N** отзывов.

**Клиентская проверка перед отправкой (вежливость, не закон):**
- Перед `POST` JS сверяет текст отзыва со словами текущей сессии — и русскими (`хромая`),
  и в раскладочной форме (`,jutvf`), без учёта регистра и разделителей.
- Если совпало — предупредить и не отправлять. Это подушка против «сделал пароль X, супер!»,
  но НЕ гарантия (ловит только слова этой сессии, произвольный пароль не детектируется).
- Смысл — остановить секрет **до** пересечения границы, в браузере.

**Серверная обработка (закон — клиент можно обойти):**
- **Экранирование при выводе** (защита от XSS) — обязательно.
- **Фильтр плохих слов** — первый барьер; дырявый (обход пробелами/leet, ложные срабатывания),
  поэтому не единственная мера.
- **Лимит частоты** (антиспам).

**Хранилище (v1):**
- Файл со **структурированными строками** (JSONL — по одному JSON-объекту на строку), чтобы
  переносы строк и спецсимволы в тексте не ломали формат.
- **Синхронизированная запись** (блокировка) — иначе одновременные отправки побьют файл.
- Чтение «последних N» — читаем файл, берём хвост.

**Модерация:**
- **Ручное удаление** — настоящий бэкстоп поверх фильтра.
- Удаление защищено (токен / защищённый эндпоинт или правка файла руками). **Не** открытая всем кнопка.

## 8. Constraints / Rules

- Случайность только криптостойкая (`crypto.getRandomValues()` на клиенте).
- Личные/секретные слова **никогда** не уходят на сервер, не логируются, не сохраняются.
- Публичный словарь: цель ≥ 400 слов (сейчас ~250); больше — лучше (каждое удвоение = +1 бит/слово).
- Leet-словарь и капитализация — публичные, в зачёт секретных слов НЕ идут.
- Отзыв: клиентский чек на секреты + серверные санитизация/лимит/фильтр обязательны;
  запись в файл синхронизирована; удаление защищено.
- Edge cases: пустой ввод, повтор слов, слово не найдено, слово целиком ушло в символы,
  пустой/слишком длинный отзыв, одновременные отправки.

## 9. Out of Scope (v1)

- Постоянное хранение паролей / встроенный «сейф».
- Аккаунты, авторизация, синхронизация между устройствами.
- Отправка пароля на email/Telegram (возвращает серверную дыру и противоречит идее
  запоминаемых паролей).
- Отзывы: без имён/регистрации, без полноценной админки модерации (только ручное удаление),
  без поиска/рейтингов по отзывам.

## 10. Future Ideas

- Другие языки и раскладки.
- Расширение публичного словаря + пополнение сообществом (серверная фича).
- **Хранилище отзывов на БД** (SQLite / Postgres через Exposed) вместо файла — заявленная
  цель обучения; даёт нормальную конкурентную запись и пагинацию «из коробки».
- Полноценная модерация/админка; рейтинги/лайки отзывов.
- Email/Telegram-доставка — только с явным предупреждением о риске и, вероятно, никогда
  для паролей с секретными словами.
- Обучающая панель: показать, как энтропия меняется от настроек.

## Files

- `words_ru.txt` — публичный словарь (сейчас ~250 слов). Отдаётся клиенту как JSON.
- `leet_ru.txt` — *(будущее)* курируемый leet-словарь (`про100`, `7я`, `о5`...).
- `reviews.jsonl` — *(серверный)* хранилище отзывов; наружу отдаётся только через API с санитизацией.
- личные секретные слова — только на клиенте, пользователь ведёт сам, в репозиторий не кладём.
